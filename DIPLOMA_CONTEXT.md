«Разработка средства автоматизации процессов анализа и оценки рисков информационной безопасности корпоративных систем банка на основе машинного обучения».

Главная задача проекта — разработать программное средство, которое автоматизирует путь от технических данных мониторинга до расчётной оценки риска и поддержки выбора мер защиты.

## Актуальная архитектурная линия

Целевая цепочка обработки:

логи / записи журналов
→ события защиты информации
→ инциденты защиты информации
→ события риска реализации информационных угроз
→ расчётная оценка риска
→ графовое уточнение
→ подбор мер защиты
→ сохранение в SQLite
→ отображение через API/UI.

Важно:
сырые логи и события защиты информации не являются событиями риска. Событие риска реализации информационной угрозы появляется только после анализа инцидента защиты информации.

## Первый классификатор

Задача:

события защиты информации → инциденты защиты информации.

Реализация:

формализованные экспертные правила + корреляция событий.

Первый классификатор не является основной ML-моделью.

Источники логики:
- ГОСТ Р 57580.1-2017 как нормативная рамка для событий защиты информации и инцидентов защиты информации;
- типовая SIEM-логика;
- корреляционные признаки;
- экспертно заданные правила прототипа;
- технические подходы detection engineering, например Sigma и MITRE ATT&CK как вспомогательная база, но не как нормативное основание.

Примеры правил:
- ошибка защитного обновления;
- отключение EDR;
- множественные неуспешные попытки аутентификации;
- обнаружение вредоносной активности;
- отказ доступа к значимому объекту;
- выгрузка данных во внерабочее время.

Пороговые значения, например failed_attempts >= 5, не являются требованием ГОСТ. Это экспертно заданные параметры прототипа.

## Второй классификатор

Задача:

инциденты защиты информации → события риска реализации информационных угроз.

Реализация:

ML-модель.

Формальная постановка:

p_s = P(y_s = 1 | X_s^risk)

где:
- y_s = 1 означает, что инцидент защиты информации отнесён к событию риска реализации информационной угрозы;
- X_s^risk — признаки инцидента, объекта защиты, актива, процесса и возможных последствий.

Признаки второго классификатора:
- тип инцидента;
- уровень инцидента;
- уверенность первого классификатора;
- количество скоррелированных событий;
- признаки ошибок аутентификации;
- признаки отключения средств защиты;
- признаки вредоносной активности;
- признаки ошибки защитного обновления;
- признаки операций с данными;
- тип объекта защиты;
- тип актива;
- сетевой сегмент;
- затронутый банковский или технологический процесс;
- возможные или фактические потери;
- признаки значимого актива;
- признаки влияния на конфиденциальность, целостность и доступность.

ML-модель не «сравнивает с ГОСТом» и не рассчитывает нормативный риск. Она автоматизирует отнесение инцидента к событию риска реализации информационной угрозы.

## Математическая модель

Главный принцип:

не переписывать математическую главу как большую формализацию логов, событий, инцидентов и событий риска.

Глава 2 должна сохранить старую основу:

- объекты защиты;
- сценарии реализации информационных угроз;
- меры защиты;
- исходная расчётная оценка риска;
- остаточная расчётная оценка риска;
- эффект мер защиты;
- бинарные переменные выбора мер;
- ограничения по бюджету и трудоёмкости;
- преобразование к максимизации эффекта;
- метод ветвей и границ.

Новая цепочка:

логи → события защиты информации → инциденты защиты информации → события риска реализации информационных угроз

должна главным образом объяснять, как формируется вероятностная составляющая p_ij^*.

Основная формула:

r_ij^0 = p_ij^* · I_ij

где:
- p_ij^* — расчётная вероятностная оценка события риска реализации сценария информационной угрозы u_j для объекта защиты o_i;
- I_ij — оценка последствий реализации сценария u_j для объекта o_i;
- r_ij^0 — исходная расчётная оценка риска.

Значимость актива и критичность процесса входят в I_ij. Нельзя использовать критичность как отдельный множитель.

Запрещённая старая формула:

risk = probability × impact × criticality

## Остаточный риск и оптимизация

Модель остаточного риска:

r_ij(x) = max(0, r_ij^0 - sum_h a_ijh x_h)

где:
- x_h — бинарная переменная выбора меры защиты;
- a_ijh — ожидаемый эффект меры h для пары «объект защиты — сценарий реализации информационной угрозы».

Целевая функция:

R(x) = sum_i sum_j r_ij(x) → min

R(x) используется только как внутренняя целевая функция модели выбора мер защиты. Нельзя писать, что это нормативный показатель совокупного риска банка.

Ограничения:

sum_h C_h x_h <= B

sum_h L_h x_h <= L_max

Опционально:

sum_h x_h <= K

Ограничение по сроку внедрения T нужно убрать из математической модели. В экономической части сроки проекта можно рассматривать как календарный план, но не как ограничение оптимизационной модели выбора мер.

## Байесовский модуль

Байесовский модуль пока отложен, но должен быть предусмотрен архитектурно.

Название:

BayesianRiskEventAdjuster

Место в конвейере:

RiskEventMLClassifier
→ BayesianRiskEventAdjuster
→ RiskEventAssessmentEngine

Назначение:

уточнение вероятностной составляющей p_ij^* на основе зависимостей между сценариями реализации информационных угроз.

Он не должен:
- заменять ML-классификатор;
- считать риск самостоятельно;
- заменять I_ij;
- становиться главным модулем проекта.

Примеры зависимостей:
- компрометация учётной записи → несанкционированный доступ;
- несанкционированный доступ → повышение привилегий;
- повышение привилегий → утечка данных;
- ошибка конфигурации → вредоносная активность.

Формулировка для текста:

Байесовский модуль используется для уточнения вероятностной составляющей расчётной оценки риска с учётом зависимостей между сценариями реализации информационных угроз. Входом модуля являются события риска реализации информационных угроз, сформированные ML-классификатором, и их базовые вероятностные оценки. Выходом является уточнённая оценка p_ij^*, используемая далее при расчёте r_ij^0 = p_ij^* · I_ij.

## Графовое уточнение

Использовать термин:

графовое уточнение расчётной оценки риска.

Не использовать как основной термин:

графовое распространение риска.

Смысл:

после локальной оценки риска учитываются инфраструктурные связи между объектами защиты. Связи с другими объектами инфраструктуры могут повышать приоритет обработки, но не должны механически раздувать риск по всей сети.

Графовое уточнение не заменяет локальную оценку.

## Подбор мер защиты

В проекте должен быть блок подбора мер обработки риска.

Смысл:

после оценки риска и графового уточнения система выбирает меры защиты, которые дают наибольшее ожидаемое снижение остаточной расчётной оценки риска при ограничениях по бюджету и трудоёмкости.

В коде новый оптимизатор должен работать с RiskAssessmentRecord, а не со старым RiskScore.

Старый ControlOptimizer можно использовать как источник идеи, но не нужно возвращать старую формулу с criticality.
