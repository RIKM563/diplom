# 2 Математическая модель оценки и обработки рисков информационной безопасности

## 2.1 Постановка задачи оценки и обработки рисков ИБ

В корпоративной системе банка оценка риска информационной безопасности не может быть сведена к изолированной обработке отдельной записи журнала или отдельного срабатывания средства защиты. Техническое событие приобретает значение для риск-ориентированного анализа только после интерпретации в контексте объекта защиты, связанного информационного актива, затрагиваемого банковского или технологического процесса, признаков возможной реализации информационной угрозы и допустимых мер обработки риска. Поэтому математическая модель должна описывать не только расчёт отдельной оценки риска, но и последовательность преобразования исходных данных в форму, пригодную для принятия решения.

В разрабатываемом программном средстве входными данными являются структурированные записи журналов и сообщения средств мониторинга: SIEM, EDR, DLP, средств сетевой защиты, операционных систем, прикладных сервисов и баз данных. Эти записи не рассматриваются как готовые события риска реализации информационных угроз. На первом этапе они нормализуются и приводятся к событиям защиты информации. Далее события защиты информации группируются и анализируются по формализованным экспертным правилам, в результате чего выделяются инциденты защиты информации. После этого инциденты обрабатываются ML-моделью, которая определяет вероятность отнесения инцидента к событию риска реализации информационной угрозы. Полученная вероятностная составляющая дополнительно уточняется с учётом зависимостей между сценариями реализации информационных угроз и далее используется при расчёте исходной оценки риска.

Для сохранения компактности модели цепочка «логи — события защиты информации — инциденты защиты информации — события риска реализации информационных угроз» не формализуется как самостоятельная большая система множеств. В рамках главы она используется прежде всего для объяснения формирования вероятностной составляющей `p_ij^*`. Основой математической постановки являются объекты защиты, сценарии реализации информационных угроз, меры защиты, исходная расчётная оценка риска, остаточная расчётная оценка риска и ограничения выбора мер защиты.

Пусть задано множество объектов защиты

`O = {o_1, o_2, ..., o_n}`,

где `o_i` обозначает элемент корпоративной банковской инфраструктуры: сервер приложений, базу данных, платёжный сервис, узел аутентификации, рабочую станцию, сетевой компонент или иной объект, для которого может быть рассчитана оценка риска. Каждый объект защиты связан с одним или несколькими информационными активами и банковскими процессами. Значимость актива, критичность процесса, влияние на конфиденциальность, целостность и доступность, а также возможные финансовые, технологические и регуляторные последствия учитываются при оценке последствий, а не вводятся как отдельный множитель риска.

Пусть также задано множество сценариев реализации информационных угроз

`U = {u_1, u_2, ..., u_m}`.

К таким сценариям относятся несанкционированный доступ, компрометация учётной записи, повышение привилегий, утечка данных, вредоносная активность, нарушение доступности сервиса, ошибка конфигурации и другие сценарии, существенные для корпоративной системы банка. Для каждой пары `(o_i, u_j)` может быть сформирована исходная расчётная оценка риска `r_ij^0`. Она отражает оценку риска до выбора дополнительных мер защиты в рамках рассматриваемой оптимизационной постановки.

Исходная расчётная оценка риска определяется через вероятностную составляющую и оценку последствий:

`r_ij^0 = p_ij^* · I_ij`,

где `p_ij^*` — уточнённая вероятностная оценка события риска реализации сценария `u_j` для объекта `o_i`, а `I_ij` — оценка последствий реализации сценария `u_j` для объекта `o_i`. Значение `p_ij^*` формируется на основе обработки исходных записей журналов, выделения событий защиты информации, выявления инцидентов защиты информации, ML-классификации событий риска и байесовского уточнения зависимых сценариев. Значение `I_ij` включает последствия для актива, бизнес- или технологического процесса, показателей конфиденциальности, целостности и доступности, а также возможные потери. В модели не вводится отдельный множитель критичности, поскольку значимость актива и критичность процесса должны входить в `I_ij`.

Для обработки риска рассматривается множество мер защиты

`H = {h_1, h_2, ..., h_q}`.

Каждая мера `h` характеризуется стоимостью внедрения `C_h`, трудоёмкостью `L_h`, областью применимости и ожидаемым эффектом для отдельных пар «объект защиты — сценарий реализации информационной угрозы». Переменная выбора меры задаётся как

`x_h ∈ {0, 1}`,

где `x_h = 1` означает, что мера `h` выбрана, а `x_h = 0` — что она не включена в набор рекомендуемых мер. Ожидаемый эффект меры `h` для пары `(o_i, u_j)` обозначается как `a_ijh`. Этот эффект интерпретируется как ожидаемое снижение расчётной оценки риска при применении меры с учётом её применимости к соответствующему сценарию и объекту.

Остаточная расчётная оценка риска после выбора мер защиты задаётся выражением

`r_ij(x) = max(0, r_ij^0 - sum_h a_ijh x_h)`.

Оператор `max` исключает получение отрицательной оценки остаточного риска. Это важно, поскольку несколько мер защиты могут быть применимы к одной паре `(o_i, u_j)`, но их суммарный ожидаемый эффект не должен приводить к содержательно невозможному отрицательному значению риска.

Задача выбора мер защиты состоит в поиске такого вектора `x`, который снижает остаточную расчётную оценку риска при ограниченных ресурсах. В качестве внутренней целевой функции используется

`R(x) = sum_i sum_j r_ij(x) -> min`.

Данная функция применяется только внутри модели выбора мер защиты и не является нормативным показателем совокупного риска банка. Она нужна для сравнения допустимых наборов мер по ожидаемому снижению расчётной оценки риска в рамках заданных объектов, сценариев и исходных данных.

Ресурсные ограничения имеют вид

`sum_h C_h x_h <= B`,

`sum_h L_h x_h <= L_max`,

где `B` — доступный бюджет, а `L_max` — допустимая трудоёмкость внедрения выбранных мер. При необходимости может использоваться дополнительное ограничение на количество выбранных мер:

`sum_h x_h <= K`.

Срок внедрения меры может храниться как справочная характеристика и использоваться при календарном планировании проекта, однако он не включается в математическую постановку оптимизационной задачи выбора мер защиты. Такой подход позволяет сохранить модель достаточно строгой и не смешивать задачу риск-ориентированного выбора мер с задачей календарного планирования работ.

Итоговая постановка объединяет два уровня. Первый уровень формирует исходную расчётную оценку риска через обработку событий, инцидентов и вероятностную оценку события риска реализации информационной угрозы. Второй уровень использует полученные оценки для выбора мер защиты при ограничениях на бюджет и трудоёмкость. В результате модель связывает технический контур обработки данных с задачей принятия решения о мерах обработки риска, сохраняя прослеживаемость от исходных записей журналов до рекомендуемого набора мер защиты.
